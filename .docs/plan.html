Đọc trước khi bắt đầu

RESP là gì? Redis nói chuyện qua TCP (cổng 6379) bằng giao thức text gọi là RESP (REdis Serialization Protocol). Client gửi mảng các bulk string, ví dụ SET foo bar đi trên dây là:

*3\r\n$3\r\nSET\r\n$3\r\nfoo\r\n$3\r\nbar\r\n

Server trả về 5 kiểu cơ bản: simple string +OK, error -ERR ..., integer :1, bulk string $5\r\nhello, hoặc array *N. Null là $-1\r\n (RESP2).

Mô hình thực thi. Redis thật chạy đơn luồng cho việc thực thi lệnh — đó là lý do mỗi lệnh là atomic. Ta tái hiện bằng coroutines: nhiều coroutine lo I/O của từng connection, nhưng việc áp dụng lệnh lên keyspace được dồn về một nơi (một coroutine/actor đọc từ Channel), nên không cần khóa (lock).

Cách test xuyên suốt: cài Redis chính chủ chỉ để lấy redis-cli (vd brew install redis), rồi chạy redis-cli -p 6379 <lệnh> trỏ vào server của bạn. Mỗi bước nên kèm vài unit test cho phần logic thuần (RESP parser, data structures).

Giai đoạn 1 · Nền tảng

Dựng được một server biết nói RESP và trả lời lệnh đầu tiên.

1

TCP accept loop với ktor-network

Lắng nghe cổng 6379, nhận nhiều client đồng thời, mỗi client một coroutine.

Việc cần làm

  • Tạo SelectorManager, mở aSocket(selector).tcp().bind("127.0.0.1", 6379).
  • Vòng lặp accept(); mỗi socket launch { handle(socket) } để xử lý song song.
  • Tạm thời: đọc từng dòng và echo lại nguyên văn (chưa parse RESP).

✦ Output cụ thể

  • Server bind 6379, log mỗi khi có client kết nối / ngắt.
  • Mở 2 terminal nc localhost 6379 cùng lúc, gõ chữ → server echo lại; chứng tỏ xử lý đa kết nối.

Kiểm thử

redis-cli -p 6379 kết nối được (chưa cần trả lời đúng). Ctrl-C tắt server gọn gàng.

2

RESP2 reader / writer

Đọc request và ghi reply đúng chuẩn RESP2 — phần lõi của giao thức.

Việc cần làm

  • sealed class RespValue: SimpleString, Error, Integer, BulkString(ByteArray?), Array(List).
  • RespReader.readCommand(): List<ByteArray>? — đọc một mảng bulk string (inline command để sau).
  • RespWriter: hàm ghi từng kiểu reply, dùng \r\n, null = $-1\r\n.
  • Giữ value ở dạng ByteArray (binary-safe), chỉ đổi sang String ở mép giao thức.

✦ Output cụ thể

  • Parse được *2\r\n$4\r\nECHO\r\n$3\r\nhey\r\n → ["ECHO","hey"].
  • Bộ unit test round-trip (đọc rồi ghi lại ra đúng bytes ban đầu) chạy xanh.

Kiểm thử

Test với input thiếu/đứt nửa chừng phải xử lý đúng (chờ thêm dữ liệu, không crash).

3

Dispatcher + PING / ECHO

Bộ điều phối lệnh và hai lệnh thật đầu tiên.

Việc cần làm

  • Map tên lệnh (uppercase) → handler; tra cứu không phân biệt hoa thường.
  • PING không tham số → +PONG; có tham số → echo tham số đó (bulk string).
  • ECHO <x> → trả x. Lệnh lạ → -ERR unknown command '...'.

✦ Output cụ thể

redis-cli ping → PONGping hi → hiecho hello → hello

Kiểm thử

redis-cli -p 6379 ping trả về PONG. Gửi lệnh bậy thấy trả (error) ERR ....

4

Keyspace + SET / GET / DEL / EXISTS

Lưu trữ thật, và mô hình single-writer để lệnh atomic.

Việc cần làm

  • class Db giữ HashMap<String, RedisObject>.
  • Mô hình đề xuất: connection coroutine gửi (command, replyChannel) vào một Channel; một coroutine thực thi tuần tự → atomic, không lock.
  • SET, GET, DEL (nhiều key), EXISTS (đếm số key tồn tại).

✦ Output cụ thể

set foo bar → OKget foo → "bar"exists foo → 1del foo → 1get foo → (nil)

Kiểm thử

Chạy redis-cli các lệnh trên theo thứ tự, kết quả khớp. Test ghi/đọc đồng thời nhiều connection không sai lệch.

5

Hết hạn (TTL / expiry)

Key tự biến mất khi hết hạn — lazy + active expiration.

Việc cần làm

  • Map phụ key → expireAtMillis. Dùng một Clock interface để dễ test (không cần sleep thật).
  • Lazy: mỗi lần truy cập key, nếu quá hạn thì xóa và coi như không tồn tại.
  • Active: coroutine định kỳ lấy mẫu ngẫu nhiên vài key hết hạn để dọn.
  • Lệnh: SET ... EX/PX, EXPIRE, PEXPIRE, TTL, PTTL, PERSIST.

✦ Output cụ thể

set k v px 100(chờ 150ms)get k → (nil)ttl k → -2

Kiểm thử

Test dùng Clock giả: set TTL, "tua" thời gian, kiểm tra key biến mất. TTL trả số giây còn lại, -1 nếu không có TTL, -2 nếu key không tồn tại.

6

Lệnh keyspace & string mở rộng

Các lệnh hay dùng để keyspace "đủ xài".

Việc cần làm

  • Keyspace: KEYS <pattern> (glob), TYPE, DBSIZE, FLUSHDB, RENAME, SCAN (cursor).
  • String/số: INCR, DECR, INCRBY, APPEND, STRLEN, GETRANGE, SETRANGE, MSET, MGET.

✦ Output cụ thể

set n 10incr n → 11type n → stringkeys * → 1) "n"

Kiểm thử

INCR trên giá trị không phải số trả -ERR ... not an integer. KEYS * liệt kê đúng các key đang sống (không tính key hết hạn).

Biến cache thành một Redis thực thụ với đủ collection types.

7

Lists (+ blocking)

Danh sách hai đầu, và lệnh chờ BLPOP — nơi coroutine tỏa sáng.

Việc cần làm

  • Dùng ArrayDeque<ByteArray> cho mỗi list.
  • RPUSH/LPUSH/LRANGE/LLEN/LINDEX/LSET/LPOP/RPOP (có count)/LREM/LTRIM.
  • Blocking BLPOP/BRPOP: nếu rỗng thì suspend chờ, đánh thức khi có push (dùng Channel hoặc danh sách waiter + CompletableDeferred).

✦ Output cụ thể

  • rpush mylist a b c → 3; lrange mylist 0 -1 → a b c.
  • Client A: blpop mylist 0 (treo chờ). Client B: rpush mylist x → Client A lập tức nhận x.

Kiểm thử

Hai redis-cli để thử BLPOP/RPUSH. Test logic chỉ số âm trong LRANGE/LINDEX.

8

Hashes

Map lồng trong key — field/value.

Việc cần làm

Mỗi hash là HashMap<String, ByteArray>. Lệnh: HSET/HGET/HMGET/HGETALL/HDEL/HEXISTS/HKEYS/HVALS/HLEN/HINCRBY/HINCRBYFLOAT.

✦ Output cụ thể

hset u name an age 20 → 2hget u name → "an"hgetall uhincrby u age 1 → 21

Kiểm thử

HGETALL trả mảng xen kẽ field/value. Gọi lệnh hash lên key kiểu string trả WRONGTYPE error.

9

Sets

Tập hợp không trùng lặp, có phép toán tập hợp.

Việc cần làm

HashSet các phần tử. Lệnh: SADD/SREM/SMEMBERS/SISMEMBER/SCARD/SPOP/SRANDMEMBER và phép toán SINTER/SUNION/SDIFF (+ biến thể ...STORE).

✦ Output cụ thể

sadd s a b c → 3sismember s a → 1scard s → 3sinter s s2

Kiểm thử

Thêm trùng không tăng kích thước. Test giao/hợp/hiệu giữa nhiều set.

10

Sorted Sets

Tập hợp có điểm số, sắp xếp — kiểu dữ liệu khó nhất.

Việc cần làm

  • Hai cấu trúc song song: HashMap<member, score> + một cây sắp theo (score, member) — TreeMap/skip list.
  • Lệnh: ZADD/ZREM/ZSCORE/ZRANK/ZREVRANK/ZRANGE (WITHSCORES)/ZREVRANGE/ZRANGEBYSCORE/ZINCRBY/ZCARD/ZCOUNT.

✦ Output cụ thể

zadd lb 100 an 90 bo → 2zrange lb 0 -1 withscoreszrank lb an → 1

Kiểm thử

Thứ tự đúng theo score rồi tới thứ tự từ điển của member khi score bằng nhau. ZRANGEBYSCORE với khoảng ( mở/đóng và -inf/+inf.

Giai đoạn 3 · Tính năng nâng cao

Pub/Sub, transactions — phần "một node feature-complete".

11

Pub / Sub

Kênh nhắn tin publish/subscribe theo thời gian thực.

Việc cần làm

  • Mỗi channel một MutableSharedFlow (hoặc danh sách subscriber).
  • SUBSCRIBE/UNSUBSCRIBE/PUBLISH, pattern PSUBSCRIBE/PUNSUBSCRIBE.
  • Khi subscribe, connection vào "chế độ subscriber" — chỉ nhận push, hạn chế lệnh thường.

✦ Output cụ thể

  • Client A: subscribe news. Client B: publish news hello → Client A nhận message news hello.
  • PUBLISH trả về số subscriber đã nhận.

Kiểm thử

Hai redis-cli thử subscribe/publish. Test pattern matching cho PSUBSCRIBE.

12

Transactions (MULTI / EXEC)

Gom nhiều lệnh chạy atomic, có optimistic lock bằng WATCH.

Việc cần làm

  • MULTI bật chế độ queue (lệnh trả +QUEUED thay vì thực thi).
  • EXEC chạy lần lượt toàn bộ queue, trả mảng kết quả. DISCARD hủy.
  • WATCH key: nếu key bị đổi trước EXEC → EXEC trả (nil) (abort). UNWATCH bỏ theo dõi.

✦ Output cụ thể

multi → OK
set a 1 → QUEUED
incr a → QUEUED
exec → 1) OK  2) (integer) 2

Kiểm thử

WATCH một key, client khác sửa key đó, rồi EXEC phải abort (trả nil). Lệnh sai cú pháp trong MULTI làm EXEC fail.

Giai đoạn 4 · Bền vững & phân tán

Persistence rồi replication — phần dài hơi nhất, làm khi nền đã chắc.

13

AOF persistence

Ghi log mọi lệnh ghi, phát lại khi khởi động.

Việc cần làm

  • Mỗi lệnh ghi (SET, DEL, LPUSH...) được append vào appendonly.aof dưới dạng RESP.
  • Khi khởi động, đọc lại file và "replay" như client gửi vào.
  • Chính sách fsync: always / everysec / no.
  • (Tùy chọn) AOF rewrite để nén file.

✦ Output cụ thể

  • Sau set foo bar, file appendonly.aof chứa lệnh đó.
  • Tắt rồi mở lại server → get foo vẫn ra bar.

Kiểm thử

Ghi vài lệnh, restart, dữ liệu còn nguyên. Mở file AOF xem nội dung là RESP hợp lệ.

14

RDB snapshot

Ảnh chụp nhị phân của toàn bộ dataset.

Việc cần làm

  • Đọc trước: parse được định dạng dump.rdb (header REDIS, version, các opcode, length encoding) — để load được file của Redis thật.
  • Ghi sau: SAVE/BGSAVE sinh dump.rdb. Load lúc khởi động (ưu tiên AOF nếu bật).

✦ Output cụ thể

  • Load được file dump.rdb do Redis chính chủ tạo → dữ liệu hiện ra qua GET/KEYS.
  • SAVE tạo file, restart đọc lại đúng.

Kiểm thử

Dùng Redis thật tạo dump nhỏ, cho server bạn đọc. So sánh keyspace trước/sau SAVE+reload.

15

Replication (master / replica)

Master truyền dòng lệnh ghi xuống các replica.

Việc cần làm

  • Cờ khởi động --replicaof <host> <port>.
  • Bắt tay từ phía replica: PING → REPLCONF listening-port / REPLCONF capa → PSYNC ? -1.
  • Master gửi +FULLRESYNC <replid> <offset> rồi truyền RDB, sau đó propagate mọi lệnh ghi.
  • Lệnh REPLICAOF, INFO replication, WAIT.

✦ Output cụ thể

  • Chạy 1 master + 1 replica. set x 1 trên master → get x trên replica trả 1.
  • INFO replication hiển thị role, số replica đang kết nối, offset.

Kiểm thử

Hai instance trên 2 cổng. Ghi master, đọc replica thấy lan truyền. Replica từ chối lệnh ghi trực tiếp (read-only).

16

Cluster (mở rộng, để sau)

Sharding theo 16384 hash slot — chỉ làm khi mọi thứ đã vững.

Việc cần làm

  • Tính slot: CRC16(key) % 16384, hỗ trợ hash tag {...}.
  • CLUSTER MYID / SLOTS / NODES / SHARDS.
  • Redirect -MOVED / -ASK khi key không thuộc node hiện tại.

✦ Output cụ thể

  • redis-cli -c truy cập key trên đúng node nhờ follow MOVED.
  • CLUSTER SLOTS trả bản đồ slot → node.

Kiểm thử

Cụm nhiều node, đặt key thuộc slot của node khác → nhận MOVED và client -c tự chuyển hướng.
